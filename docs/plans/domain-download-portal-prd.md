# PRD: 域名正规化与下载门户建设

## Problem Statement

当前 PhoneTalk 的连接与分发流程存在以下问题：

1. **裸域名使用**：现有代码中 `ilovin.xyz` 硬编码于多处（`install.sh`、`agentapp` 默认服务器、`agentgw` 默认隧道地址）。该域名未来可能承载其他服务，存在路由冲突和扩展性风险。
2. **无统一域名体系**：隧道中继、下载站、API 服务没有清晰的命名空间划分，不利于未来功能扩展（如新增应用服务、文档站点等）。
3. **无公网下载站**：外部用户只能通过本地 `agentgw`（`http://localhost:7374/apk`）获取客户端，无法在未运行网关的设备上完成首次安装。本地 Flutter Web 管理页也无法从公网访问。
4. **配置不幂等**：`install.sh` 等脚本没有域名变量化能力，换域名需要全局搜索替换，容易遗漏且无法支持多环境构建。
5. **缺乏用户侧诊断数据**：当用户遇到连接问题时，没有机制收集客户端版本、平台、运行状态和日志，导致远程排查困难。

## Solution

建立以 `ilovim.xyz` 为根域的功能子域名体系，将隧道中继、下载站、API 服务隔离到独立子域。实现域名编译时注入+运行时覆盖的混合配置模式，确保开箱即用且支持灵活切换。搭建公网下载门户（静态交互向导页 + 轻量 API），让外部用户无需预先安装任何组件即可下载客户端、获取安装指引和连接配置。新增 telemetry 上报机制，在用户授权下收集诊断信息以支持问题排查。

## User Stories

1. 作为外部用户，我希望能通过浏览器访问一个公网网站来下载 PhoneTalk 客户端，这样我就不需要先运行本地网关才能获取 APK。
2. 作为 Mac 用户，我希望能复制一条 `curl | sh` 命令完成一键安装，这样我就不需要手动下载二进制、配置权限和生成 token。
3. 作为 Linux 服务器用户，我希望安装脚本能自动检测我的平台架构并安装正确的产物，这样我就不需要手动区分 amd64/arm64。
4. 作为已安装用户，我希望运行同一个安装脚本就能检测到现有安装并执行增量更新，这样我就不需要卸载重装。
5. 作为开发者，我希望保留本地 `localhost:7374/apk` 的下载能力，这样我在内网开发测试时不受公网站点可用性影响。
6. 作为运维人员，我希望域名可以在编译时通过环境变量注入，这样我可以用同一套代码为不同环境（dev/staging/prod）构建产物。
7. 作为普通用户，我希望域名升级时客户端能自动跟随新域名，这样我就不需要手动修改配置文件。
8. 作为 Android 用户，我希望在下载站上扫描一个 QR 码就能直接下载 APK，这样我就不需要手动输入 URL。
9. 作为遇到连接问题的用户，我希望能点击"提交诊断报告"按钮将日志和状态发送给开发团队，这样我能更快获得帮助。
10. 作为支持人员，我希望能通过 telemetry 数据看到用户端的版本、平台和组件运行状态，这样我能快速定位问题。
11. 作为项目维护者，我希望 release 脚本能自动将产物上传到 OSS 并刷新 CDN，这样新版本发布后用户能立即下载。
12. 作为未来功能扩展者，我希望使用 `app.ilovim.xyz` 这样的子域名来部署新应用，这样我就不会和现有的隧道/下载服务发生冲突。
13. 作为安全负责人，我希望下载站有基本的访问日志和限流，这样能防止恶意刷流量导致账单暴涨。
14. 作为首次安装用户，我希望安装完成后能看到清晰的下一步指引（如何启动、如何连接手机），这样我就不会安装完不知所措。
15. 作为远程模式用户，我希望安装脚本生成的默认配置包含正确的隧道域名，这样我启动后就能直接连接到隧道中继。
16. 作为局域网模式用户，我希望安装脚本自动检测本机 IP 并生成 LAN 连接配置，这样我手机扫码时无需手动输入 IP。
17. 作为项目维护者，我希望 release manifest 包含所有平台产物的 SHA256 校验值，这样客户端更新时能验证下载完整性。
18. 作为 API 消费者，我希望通过 `/v1/release/latest` 获取结构化的版本元数据，这样我能编写自动更新检查逻辑。
19. 作为 iOS 开发者，我希望代码中保留 iOS 构建能力，但公网下载站明确标注 iOS 暂不提供，这样不会误导普通用户。
20. 作为 tunnelhub 运维，我希望节点状态查询保留在后台（不暴露公网），这样既能排查问题又不泄露用户拓扑信息。

## Implementation Decisions

### 域名体系与子域名分配

采用功能子域名化风格（Google-style），根域为 `ilovim.xyz`，默认通过编译时注入，各功能隔离如下：

- `tunnel.ilovim.xyz`：WebSocket/gRPC-Web 隧道中继，替代现有 `ilovin.xyz`。
- `download.ilovim.xyz`：静态交互向导页，CDN 加速，公开访问。
- `api.ilovim.xyz`：轻量后端 API，负责版本查询、install.sh 分发、telemetry 接收。

### 域名变量化（混合模式）

- **编译时注入**：`scripts/build.sh` 读取 `DOMAIN` 环境变量，通过 Go `-ldflags` 和 Flutter `--dart-define` 注入默认值。产物构建后域名即固定，确保开箱即用。
- **运行时覆盖**：各组件启动时读取环境变量（如 `PHONETALK_DOMAIN`）或配置文件（`~/.agentgw/config.json` 中的 `tunnelDomain`/`apiDomain`），覆盖编译时默认值。
- **自动跟随**：当默认域名变更时，重新编译发布新版本，用户通过自动更新获取新域名，无需手动干预。

### Release Manifest 标准化

`scripts/release.sh` 生成标准化的 `manifest.json`，包含：
- 版本号、发布时间
- 按平台/架构分组的产物列表（agentd、agentgw、apk、ipa）
- 每个产物的下载 URL 和 SHA256 校验值
- 产物存储于阿里云 OSS / 腾讯云 COS，通过 CDN 分发

`scripts/release.sh --publish` 自动上传产物到 OSS 并刷新 CDN 缓存。

### 一键安装脚本

`api.ilovim.xyz/v1/install.sh` 根据 `User-Agent` 返回对应平台脚本，核心能力：
- **平台检测**：解析 `uname -s` 和 `uname -m`，映射到 `darwin-arm64`、`linux-amd64` 等目标三元组。
- **全量安装**：同时安装 `agentd` 和 `agentgw`（远程服务器也必须安装网关才能建立隧道和提供显示）。
- **幂等配置**：首次运行时生成随机 token 并写入 `~/.agentgw/config.json`；后续运行读取现有配置，不覆盖用户修改。
- **增量更新**：`install.sh --update` 对比本地版本与 `/v1/release/latest`，仅下载变更产物并覆盖，保留配置。
- **安装后指引**：输出下一步命令（`agentgw start --qr`）和下载站 URL（获取手机 App）。

### 下载站页面（静态交互向导）

`download.ilovim.xyz` 为静态站点（Cloudflare Pages / 国内对应服务），前端交互：
- 场景选择卡片：Mac 本机 / Linux 服务器 / Android
- 根据选择展示对应内容：一键安装命令、APK 下载 QR 码、连接指引
- 通用说明：远程连接地址模板、启动后终端 QR 码查看方式
- 不提供实时节点状态（该功能保留在 tunnelhub 后台）
- iOS 入口隐藏或标注"开发者专用"

### Telemetry 模块

新增可测试的 telemetry 客户端模块，封装上报逻辑：
- **触发时机**：agentgw 启动时上报一次；agentd/agentgw 异常退出时自动上报最后日志；用户通过 `agentgw debug --submit` 主动触发。
- **上报内容**：installId（随机 UUID）、版本、平台、运行时长、各组件状态（agentgw/agentd 是否运行）、网络模式（LAN/Tunnel）、脱敏后的日志片段。
- **用户关联**：携带 OpenSSO accessToken，服务端解析用户 ID 后入库，支持按用户追溯问题。
- **隐私边界**：只收集日志，不上传 `config.json` 等配置文件，避免 token 等敏感信息泄露。

### API 服务

`api.ilovim.xyz` 为独立轻量服务（建议部署于现有 K8s 集群），端点：
- `GET /v1/release/latest`：返回最新版本元数据和全平台产物下载链接。
- `GET /v1/install.sh`：根据 `User-Agent` 返回平台特定的安装脚本。
- `GET /v1/health`：服务健康检查。
- `POST /v1/telemetry`：接收客户端状态/日志上报。

### 防刷机制

- 所有端点记录 `X-Real-IP`、`User-Agent`、时间戳到日志。
- 静态资源和下载链接通过 CDN 边缘限流。
- API 端点按 IP 做请求频率限制（如 60 请求/分钟）。

## Testing Decisions

### 什么是一个好的测试

- 只测试外部行为（输入/输出），不测试实现细节（如内部函数调用顺序）。
- 优先测试边界条件和失败路径（网络超时、平台检测失败、幂等配置冲突）。
- 对于安装脚本，使用容器化环境模拟不同 `uname` 输出，验证产物选择和下载 URL 生成。

### 需要测试的模块

1. **Release Manifest 模块**：验证 `manifest.json` 生成逻辑，确保所有平台产物都有 URL 和 SHA256。
2. **平台检测模块**：验证 `install.sh` 中 `uname` 到目标三元组的映射矩阵（Darwin/Linux × arm64/amd64）。
3. **幂等配置模块**：验证首次生成 token 与后续读取现有配置的流程，确保重复执行不覆盖用户自定义值。
4. **Telemetry 客户端模块**：验证上报 payload 的结构化、日志脱敏、触发条件（启动/异常/主动）。
5. **域名解析模块**：验证编译时默认值、环境变量覆盖、配置文件覆盖的优先级顺序。

### 现有测试参考

- `agentgw/internal/ws/server_test.go`：WebSocket 服务端测试模式可参考用于 API 服务端点测试。
- `agentd/internal/agent/manager_test.go`：状态机管理测试模式可参考用于 telemetry 组件状态上报测试。

## Out of Scope

1. **实时节点状态仪表盘**：用户节点在线状态查询保留在 tunnelhub 后台，不暴露到公网 API。
2. **iOS 公网分发**：TestFlight/企业证书/OTA 分发不在本次范围内，iOS 继续使用 Xcode 本地刷入。
3. **多语言国际化**：下载站和安装脚本先以中文为主，英文支持后续迭代。
4. **P2P 连接模式**：Tailscale/内网穿透等直连模式保持现有实现，不在本次改造范围内。
5. **Web 端使用**：Flutter Web 的公网托管不在本次范围内（仍由本地 agentgw 在 `localhost:7374` 提供）。
6. **付费/配额系统**：OSS 流量和 CDN 成本监控由现有云基础设施告警覆盖，不在应用层实现。

## Further Notes

1. **现有 `ilovin.xyz` 的迁移**：代码库中 `ilovin.xyz` 需全局替换为变量引用。建议保留 `ilovin.xyz` 的 DNS 解析一段时间（如 3 个月），做 301 重定向到 `tunnel.ilovim.xyz`，避免已安装用户断连。
2. **开发者模式保留**：本地 `agentgw` 的 `http://localhost:7374/apk` 和 QR 码打印能力继续保留，供内网开发测试使用。公网下载站仅面向外部最终用户。
3. **产物存储选择**：如果团队已有阿里云/腾讯云账号，优先使用其 OSS + CDN；否则 GitHub Releases 也可作为起步方案（国内访问质量需评估）。
4. **APK 签名**：公网分发的 APK 必须使用发布证书签名（不能是 debug 签名），否则 Android 系统会阻止安装。
5. **安装脚本安全**：`curl | sh` 模式存在供应链攻击风险。建议在下载站同时提供"手动下载二进制"的替代路径，并在脚本中加入 checksum 验证。
